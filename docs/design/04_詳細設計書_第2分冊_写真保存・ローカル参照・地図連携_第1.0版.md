# 04_詳細設計書_第2分冊_写真保存・ローカル参照・地図連携

版数: 第1.0版

## 本書の位置づけ

本書は、共通基本原則に従い、写真保存、OCR主写真、ローカル参照、地図連携を定義する。写真本体はBase64文字列を正本にせず、`FieldTest.TestPhoto`には保存先、ハッシュ、写真種別、OCR主写真フラグ等のメタデータを保持する。

## 1. 写真保存方針

写真本体はBase64文字列を正本にしない。クラウド側ではBlobまたはファイルストレージ、ローカル側では管理対象フォルダまたは既定保存先を使用し、`FieldTest.TestPhoto`には保存先URI、ローカルパス、SHA-256、MIME type、サイズ、撮影日時、写真種別を保持する。

## 2. OCR主写真

同一試験結果に対して`is_ocr_primary = 1`の写真は1件のみ許可する。DB側ではfiltered unique indexを使用し、アプリ制御だけに依存しない。

## 3. 写真同期

1. 現場Webで写真を登録する。
2. クラウドAPIが写真本体をストレージに保存し、ハッシュを計算する。
3. OCR対象写真はOCR処理へ送信する。
4. Sync Agentは写真メタデータと保存先を受信し、Providerにかかわらず`FieldTest.TestPhoto`へ登録する。
5. 再送時は`event_id`または`sha256 + result_id + photo_kind`で重複を防ぐ。

## 4. ローカル参照

Providerアプリケーションがローカルで写真を参照する場合は、`FieldTest.TestPhoto.storage_uri`または`local_file_path`を参照する。Sync AgentはFieldTest登録までを責務とし、Providerアプリケーション側の表示方法には依存しない。

## 5. 地図連携

現場マスターに緯度・経度が存在する場合はDTOに含める。存在しない場合は住所・現場名での検索候補に留め、Sync Agentが外部地図APIへ直接問い合わせることは必須としない。


## 7. 写真の重複防止

`FieldTest.TestPhoto`は、同一`event_id`の再送だけでなく、同一`result_id`、同一`photo_kind`、同一`sha256`の再送を重複として扱う。写真メタデータには、`storage_uri`または`local_file_path`のいずれかを必須とし、写真所在を持たないレコードは登録しない。

## 8. OCR主写真の一意性

同一`result_id`に対して`is_ocr_primary = 1`の写真は1件のみとする。OCR主写真を差し替える場合は、旧主写真の`is_ocr_primary`を0に更新したうえで新主写真を登録する。履歴として写真レコードは削除しない。
