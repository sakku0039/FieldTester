# 01_要件定義書

版数: 第1.0版

## 設計原則

- Sync Agent設定でProviderを明示選択し、`Labonity`または`Liberty`として起動する。Providerの自動判定、別Providerへの自動フォールバック、サンプルDB名による推測接続は行わない。
- 両Providerとも、入口設定ファイルまでの完全パスをSync Agent設定で指定する。Labonityでは`LibLocal.xml`、Libertyでは`LIBLocal.Ini`を入口設定ファイルとする。
- Labonity Providerでは、`LibLocal.xml`の`p_サーバーフォルダパス`から`LibertyDatabaseSetting.xml`を解決し、SQL Server名、マスターDB、出荷管理DB、共通設定DBを取得する。`p_SQLサーバー名`が未設定または空白の場合は設定エラーとして停止し、`localhost`、`.`、端末名、サンプル値、Liberty側設定値への補完やfallbackは行わない。
- Liberty Providerでは、`LIBLocal.Ini`の`[端末設定] サーバーパス`から`LIBCTRL.ini`を解決し、SQL Server名、共通マスタDB、出荷データDB、初期工場コードを取得する。`ＳＱＬサーバー名`が未設定または空白の場合も設定エラーとして停止する。
- 第1.0版OCR同期における必須DBは、出荷DBとマスターDBである。品質管理DB、品管データDB、品管データ2DB、写真台帳DB、販売DB、受払DB、動荷重DBは、設定ファイルに存在しても同期対象・必須接続先・登録先に含めない。
- 共通化する対象は、Sync Agent内部DTO、クラウド同期API、監査、認証、同期状態、出荷DB内の`FieldTest`登録スキーマである。読取元の既存DB差異はProvider Adapterが吸収する。
- `FieldTest`スキーマ名は第1.0版では固定であり、Provider設定による任意名変更は行わない。DDL、Migration、Providerアプリ参照View、受入条件はすべて固定名`FieldTest`を正とする。
- 既存出荷レコードを一意に表す`source_shipment_key`と、試験結果を一意に表す`source_result_key`は、Provider別に定義したcanonical JSONから生成し、キー本文、キーversion、SHA-256を保存する。
- 写真本体はBase64文字列を正本にせず、クラウド・ローカルともにバイナリまたはファイル/Blob参照として扱う。`FieldTest.TestPhoto`には写真メタデータ、ハッシュ、保存先、OCR主写真フラグを保持する。
- 秘密値は設定ファイルから読み取っても、ログ、監査、画面、設計資料に平文出力しない。サンプルではパスワード、実IP、ユーザー名、会社情報をマスク可能な値として扱う。

## 1. 目的

現場試験アプリは、出荷予定、出荷実績、現場情報を現場で参照し、フレッシュ試験結果、写真、OCR確認結果を確定登録するためのアプリケーションである。Sync AgentはクラウドAPIとProvider別既存DBの間に配置し、読取元の差異を吸収しながら、登録先を両Provider共通の`FieldTest`スキーマに統一する。

## 2. 対象Provider

|項目|Labonity Provider|Liberty Provider|
|---|---|---|
|選択値|`Labonity`|`Liberty`|
|入口設定|Sync Agent設定で指定した`LibLocal.xml`|Sync Agent設定で指定した`LIBLocal.Ini`|
|派生設定|`LibLocal.xml`のサーバーフォルダ配下の`LibertyDatabaseSetting.xml`|`LIBLocal.Ini`のサーバーパス配下の`LIBCTRL.ini`|
|SQL Server|`p_SQLサーバー名`。空欄は設定エラー|`ＳＱＬサーバー名`。空欄は設定エラー|
|マスターDB|`p_マスターデータベース名`。代表値`MSDAT`|`共通マスタＤＢ名`。代表値`LMASTER`|
|出荷DB|`p_出荷管理データベース名`。代表値`NADAT`|`出荷データＤＢ名`。代表値`LSYUDAT`|
|設定DB|`p_共通設定データベース名`。代表値`LIBERTYSETTINGS`|`LIBCTRL.ini`のシステム設定|
|結果登録|出荷DB内の`FieldTest`スキーマ|出荷DB内の`FieldTest`スキーマ|
|写真登録|`FieldTest.TestPhoto`にメタデータ、写真本体はファイル/Blob参照|`FieldTest.TestPhoto`にメタデータ、写真本体はファイル/Blob参照|
|品質系DB|第1.0版OCR同期では非使用|第1.0版OCR同期では非使用|
|Providerアプリ側連携|Labonity側アプリケーションが`FieldTest`を参照する|Liberty側アプリケーションが`FieldTest`を参照する|

## 3. 機能要件

|ID|要件|内容|
|---|---|---|
|REQ-001|Provider選択|Sync Agent設定で`Labonity`または`Liberty`を明示選択できる。|
|REQ-002|Provider固定|設定不備や接続失敗時に別Providerへ自動切替しない。|
|REQ-003|入口パス設定|Provider別入口設定ファイルまでの完全パスをSync Agent設定に保持できる。|
|REQ-004|Labonity入口解決|`LibLocal.xml`から`p_サーバーフォルダパス`を取得し、`LibertyDatabaseSetting.xml`を解決できる。|
|REQ-005|Labonity DB解決|`LibertyDatabaseSetting.xml`からSQL Server名、マスターDB、出荷管理DB、共通設定DBを取得できる。|
|REQ-006|Liberty入口解決|`LIBLocal.Ini`から`サーバーパス`を取得し、`LIBCTRL.ini`を解決できる。|
|REQ-007|Liberty DB解決|`LIBCTRL.ini`からSQL Server名、共通マスタDB、出荷データDB、初期工場コードを取得できる。|
|REQ-008|非使用DB制御|品質系DB、写真台帳DB、販売DB等は第1.0版OCR同期の必須接続先にしない。|
|REQ-009|既存予定読取|Provider別の予定データを共通DTOへ変換できる。|
|REQ-010|既存出荷読取|Provider別の出荷実績データを共通DTOへ変換できる。|
|REQ-011|現場読取|Provider別の現場マスターを共通DTOへ変換できる。|
|REQ-012|共通FieldTest登録|両Providerとも、解決した出荷DB内の`FieldTest`スキーマへ結果、写真、OCR、同期状態を登録できる。|
|REQ-013|Libertyアプリ連携|Liberty側アプリケーションは`FieldTest`スキーマを参照して結果を取得する。Sync Agentは参照画面の実装に依存しない。|
|REQ-014|Migration|`FieldTest`スキーマ作成、差分検査、再実行安全、SchemaMigration記録を行える。|
|REQ-015|写真保存|写真本体はBase64文字列を正本にせず、ファイル/Blob参照とメタデータで管理できる。|
|REQ-016|OCR主写真一意|同一試験結果に対して現在有効なOCR主写真を1件に制約できる。|
|REQ-017|共通DTO|クラウドAPIはProvider差異を持たず、Sync Agent AdapterがDTO変換を担う。|
|REQ-018|同期API|写真、試験結果、OCR結果、設定状態、監査イベントを同期できる。|
|REQ-019|オフライン|通信断時にローカルキューへ蓄積し、復旧時に順序性を保って送信できる。|
|REQ-020|再送安全|同一eventIdを再送しても重複登録しない。|
|REQ-021|監査|設定解決、読取、登録、同期、更新、エラーを監査できる。|
|REQ-022|自動更新|署名とSHA-256を検証してSync Agentを自動更新できる。|
|REQ-023|ロールバック|更新失敗時に前回安定版へ戻せる。|
|REQ-024|秘密値保護|SQLパスワード等をログ、監査、画面、資料に平文出力しない。|
|REQ-025|権限分離|利用者アカウントとAgent credentialを分離できる。|
|REQ-026|ID分離|plantIdはGUID、plantCodeはProvider側工場コードとして分離できる。|
|REQ-027|source key保持|Provider別の出荷識別キーを`FieldTest.SourceShipmentMap`と`source_shipment_key`に保持できる。|
|REQ-028|構成情報非依存|サンプルDB名を固定値として埋め込まず、設定ファイルの値を正とする。|
|REQ-029|受入追跡|要件と受入条件をExcelで追跡できる。|
|REQ-030|Providerアプリ境界|既存Providerアプリ側の参照実装はProviderアプリ側範囲とし、Sync AgentはFieldTest登録までを担当する。|

## 4. 非機能要件

|区分|要件|
|---|---|
|可用性|Sync Agent停止時も既存DB正本を破壊しない。再起動後に差分同期できる。|
|性能|出荷一覧検索は通常利用範囲で数秒以内を目標とし、大量写真同期はバックグラウンド処理にする。|
|セキュリティ|Agent credentialは個人アカウントと分離し、秘密値は暗号化保管・マスク表示する。|
|監査|設定解決、読取、登録、削除、同期、更新、例外を追跡可能にする。|
|移行性|DB名はサンプル値ではなく設定ファイル由来値を正とする。|
|保守性|Provider別差異はProvider Adapterに閉じ込め、クラウドAPIと現場WebにProvider分岐を持ち込まない。|
|データ分離|登録結果は既存出荷DB内の`FieldTest`スキーマで分離し、既存出荷テーブルを直接変更しない。|

## 5. 範囲外

TP採取、供試体、強度試験、品質帳票DBへの登録は第1.0版OCR同期の範囲外とする。Providerアプリケーションが`FieldTest`スキーマをどの画面や帳票で参照するかはProviderアプリケーション側の実装範囲であり、Sync Agentは`FieldTest`への正規登録、同期状態、監査までを担当する。


## 7. 実装詳細要件

|ID|要件|内容|
|---|---|---|
|REQ-031|SQL Server名必須|Labonityの`p_SQLサーバー名`、Libertyの`ＳＱＬサーバー名`は空欄不可とする。空欄の場合は`CFG-DB-001`として停止し、補完・推測・fallbackは行わない。|
|REQ-032|FieldTest固定|第1.0版の登録スキーマ名は`FieldTest`固定とし、設定項目で変更しない。|
|REQ-033|source key正規化|既存出荷キー、試験キーはProvider別canonical JSON、version、SHA-256で保持する。|
|REQ-034|自然キー冪等性|同一eventIdだけでなく、同一source_result_keyの再送でも重複登録しない。|
|REQ-035|写真冪等性|同一結果・同一写真種別・同一SHA-256の写真は別eventIdでも重複登録しない。|
|REQ-036|Inbound管理|クラウドから受信したイベントは`FieldTest.InboundEvent`で処理状態、retry、エラーを管理する。|
|REQ-037|Outbox管理|Provider DBからクラウドへ送信するイベントは`FieldTest.OutboxEvent`で送信状態、retry、ACKを管理する。|
|REQ-038|Provider参照View|Providerアプリ側は`FieldTest.v_ProviderApp_*`を参照できる。Sync AgentはView提供までを担当し、画面実装には依存しない。|
|REQ-039|Migration再実行安全|Migrationはversion単位で記録し、列・制約・indexの互換性を検査したうえで適用する。|
|REQ-040|Lease再取得|期限切れAgent leaseは同一トランザクションで解放扱いにし、競合なく再取得できる。|
|REQ-041|黒板・TP境界|第1.0版OCR同期では黒板テンプレート生成、TP採取、供試体、強度試験、品質帳票DBへの登録を含めない。|
|REQ-042|数値保持|OCR元文字列と確定数値を分離し、帳票・範囲検査・集計に使用できる。|
|REQ-043|設定診断エラーコード|設定、DB、Migration、同期、権限のエラーコードを分類して監査・診断に保存する。|
|REQ-044|秘密値保護詳細|SQLパスワード、接続文字列、Agent secret、tokenをログ・監査・画面に平文出力しない。|

## 8. 受入条件の責務分離

受入条件は、Sync Agentが合格判定できるものと、Providerアプリ側改造完了後に合格判定するものを分離する。Sync Agentは設定解決、DTO変換、FieldTest登録、同期、監査、View提供までを責務とする。Providerアプリは、`FieldTest.v_ProviderApp_*`を参照した表示・帳票・検索を責務とする。
