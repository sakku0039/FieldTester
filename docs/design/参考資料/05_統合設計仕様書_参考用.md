# 99_統合設計仕様書

版数: 第1.0版

本書は、現場試験アプリ第1.0版の設計資料を統合した仕様書である。各資料は初版仕様として記載し、Provider設定、DB解決、FieldTest統一登録、同期API、Migration、受入条件を一体として定義する。


## 共通基本原則

- Sync Agent設定でProviderを明示選択し、`Labonity`または`Liberty`として起動する。Providerの自動判定、別Providerへの自動フォールバック、サンプルDB名による推測接続は行わない。
- 両Providerとも、入口設定ファイルまでの完全パスをSync Agent設定で指定する。Labonityでは`LibLocal.xml`、Libertyでは`LIBLocal.Ini`を入口設定ファイルとする。
- Labonity Providerでは、`LibLocal.xml`の`p_サーバーフォルダパス`から`LibertyDatabaseSetting.xml`を解決し、SQL Server名、マスターDB、出荷管理DB、共通設定DBを取得する。`p_SQLサーバー名`が未設定または空白の場合は設定エラーとして停止し、`localhost`、`.`、端末名、サンプル値、Liberty側設定値への補完やfallbackは行わない。
- Liberty Providerでは、`LIBLocal.Ini`の`[端末設定] サーバーパス`から`LIBCTRL.ini`を解決し、SQL Server名、共通マスタDB、出荷データDB、初期工場コードを取得する。`ＳＱＬサーバー名`が未設定または空白の場合も設定エラーとして停止する。
- 第1.0版OCR同期における必須DBは、出荷DBとマスターDBである。品質管理DB、品管データDB、品管データ2DB、写真台帳DB、販売DB、受払DB、動荷重DBは、設定ファイルに存在しても同期対象・必須接続先・登録先に含めない。
- 共通化する対象は、Sync Agent内部DTO、クラウド同期API、監査、認証、同期状態、出荷DB内の`FieldTest`登録スキーマである。読取元の既存DB差異はProvider Adapterが吸収する。
- `FieldTest`スキーマ名は第1.0版では固定であり、Provider設定による任意名変更は行わない。DDL、Migration、Providerアプリ参照View、受入条件はすべて固定名`FieldTest`を正とする。
- 既存出荷レコードを一意に表す`source_shipment_key`と、試験結果を一意に表す`source_result_key`は、Provider別に定義したcanonical JSONから生成し、キー本文、キーversion、SHA-256を保存する。
- 写真本体はBase64文字列を正本にせず、クラウド・ローカルともにバイナリまたはファイル/Blob参照として扱う。`FieldTest.TestPhoto`には写真メタデータ、ハッシュ、保存先、OCR主写真フラグを保持する。
- 秘密値は設定ファイルから読み取っても、ログ、監査、画面、設計資料に平文出力しない。サンプルではパスワード、実IP、ユーザー名、会社情報をマスク可能な値として扱う。

## 1. 目的

現場試験アプリは、出荷予定、出荷実績、現場情報を現場で参照し、フレッシュ試験結果、写真、OCR確認結果を確定登録するためのアプリケーションである。Sync AgentはクラウドAPIとProvider別既存DBの間に配置し、読取元の差異を吸収しながら、登録先を両Provider共通の`FieldTest`スキーマに統一する。

## 02_基本設計書


版数: 第1.0版

### 設計原則

- Sync Agent設定でProviderを明示選択し、`Labonity`または`Liberty`として起動する。Providerの自動判定、別Providerへの自動フォールバック、サンプルDB名による推測接続は行わない。
- 両Providerとも、入口設定ファイルまでの完全パスをSync Agent設定で指定する。Labonityでは`LibLocal.xml`、Libertyでは`LIBLocal.Ini`を入口設定ファイルとする。
- Labonity Providerでは、`LibLocal.xml`の`p_サーバーフォルダパス`から`LibertyDatabaseSetting.xml`を解決し、SQL Server名、マスターDB、出荷管理DB、共通設定DBを取得する。`p_SQLサーバー名`が未設定または空白の場合は設定エラーとして停止し、`localhost`、`.`、端末名、サンプル値、Liberty側設定値への補完やfallbackは行わない。
- Liberty Providerでは、`LIBLocal.Ini`の`[端末設定] サーバーパス`から`LIBCTRL.ini`を解決し、SQL Server名、共通マスタDB、出荷データDB、初期工場コードを取得する。`ＳＱＬサーバー名`が未設定または空白の場合も設定エラーとして停止する。
- 第1.0版OCR同期における必須DBは、出荷DBとマスターDBである。品質管理DB、品管データDB、品管データ2DB、写真台帳DB、販売DB、受払DB、動荷重DBは、設定ファイルに存在しても同期対象・必須接続先・登録先に含めない。
- 共通化する対象は、Sync Agent内部DTO、クラウド同期API、監査、認証、同期状態、出荷DB内の`FieldTest`登録スキーマである。読取元の既存DB差異はProvider Adapterが吸収する。
- `FieldTest`スキーマ名は第1.0版では固定であり、Provider設定による任意名変更は行わない。DDL、Migration、Providerアプリ参照View、受入条件はすべて固定名`FieldTest`を正とする。
- 既存出荷レコードを一意に表す`source_shipment_key`と、試験結果を一意に表す`source_result_key`は、Provider別に定義したcanonical JSONから生成し、キー本文、キーversion、SHA-256を保存する。
- 写真本体はBase64文字列を正本にせず、クラウド・ローカルともにバイナリまたはファイル/Blob参照として扱う。`FieldTest.TestPhoto`には写真メタデータ、ハッシュ、保存先、OCR主写真フラグを保持する。
- 秘密値は設定ファイルから読み取っても、ログ、監査、画面、設計資料に平文出力しない。サンプルではパスワード、実IP、ユーザー名、会社情報をマスク可能な値として扱う。

## 1. 全体構成

現場WebはクラウドAPIへアクセスし、クラウドAPIはSync Agentと同期する。Sync AgentはProvider設定に従ってLabonityまたはLibertyの入口設定ファイルを読み取り、実DB名を解決する。DB解決後はProvider Adapterが既存テーブルと共通DTOの変換を担当し、登録結果はProviderにかかわらず出荷DB内の`FieldTest`スキーマへ保存する。

## 2. コンポーネント

|コンポーネント|役割|
|---|---|
|現場Web|出荷検索、試験結果入力、写真登録、OCR確認を行う。|
|クラウドAPI|認証、同期API、写真アップロード、OCR、監査、自動更新manifestを提供する。|
|Sync Agent|Provider設定解決、既存DB読取、FieldTest登録、クラウド同期、自動更新を行う。|
|Labonity Adapter|`LibLocal.xml`と`LibertyDatabaseSetting.xml`を読み、Labonity DBをDTOへ変換する。|
|Liberty Adapter|`LIBLocal.Ini`と`LIBCTRL.ini`を読み、Liberty DBをDTOへ変換する。|
|FieldTestスキーマ|試験結果、写真メタデータ、OCR結果、同期状態、監査補助を保持する。|

## 3. Provider別データ配置

|項目|Labonity Provider|Liberty Provider|
|---|---|---|
|選択値|`Labonity`|`Liberty`|
|入口設定|Sync Agent設定で指定した`LibLocal.xml`|Sync Agent設定で指定した`LIBLocal.Ini`|
|派生設定|`LibLocal.xml`のサーバーフォルダ配下の`LibertyDatabaseSetting.xml`|`LIBLocal.Ini`のサーバーパス配下の`LIBCTRL.ini`|

|出荷DB|`p_出荷管理データベース名`。代表値`NADAT`|`出荷データＤＢ名`。代表値`LSYUDAT`|
|設定DB|`p_共通設定データベース名`。代表値`LIBERTYSETTINGS`|`LIBCTRL.ini`のシステム設定|

|品質系DB|第1.0版OCR同期では非使用|第1.0版OCR同期では非使用|
|Providerアプリ側連携|Labonity側アプリケーションが`FieldTest`を参照する|Liberty側アプリケーションが`FieldTest`を参照する|

## 4. データフロー

1. Sync Agentが起動し、Sync Agent設定のProvider値を読む。
2. Providerに応じて入口設定ファイルの完全パスを検証する。
3. Labonityでは`LibLocal.xml`から`LibertyDatabaseSetting.xml`、Libertyでは`LIBLocal.Ini`から`LIBCTRL.ini`を解決する。
4. SQL Server名、出荷DB名、マスターDB名、工場コード、共通設定DB名を取得する。
5. 出荷DBに`FieldTest` Migrationを適用または差分診断する。
6. 既存DBから予定・出荷・現場情報を読み取り、共通DTOへ変換する。
7. 現場Webから登録された試験結果・写真・OCR結果をクラウド経由またはローカルキュー経由でSync Agentが受け取る。
8. Sync AgentはProviderにかかわらず、出荷DBの`FieldTest.TestResult`、`FieldTest.TestPhoto`、`FieldTest.OcrResult`へ登録する。
9. 登録結果と同期状態を監査し、必要に応じてクラウドへ再送する。
10. Providerアプリケーションは、各Providerの出荷DB内`FieldTest`スキーマを参照して結果を利用する。

## 5. ID体系

|ID|内容|
|---|---|
|tenantId|クラウドテナント識別子。|
|orgId|組織識別子。|
|plantId|クラウド・Sync Agentで使う工場GUID。|
|plantCode|Provider別の工場コード。Labonityは設定値またはDB値、Libertyは`初期工場コード`を既定値とする。|
|sourceShipmentKey|Provider別の出荷識別キー。既存DBの主キーまたは複合キーを文字列化して保持する。|
|eventId|クラウド・Sync Agent間の冪等性を保証するイベントID。|

## 6. DB名解決の考え方

サンプルDB名は設計上の代表名であり、実行時は設定ファイルの値を正とする。設定ファイルに値が存在しない場合は、Sync Agent設定で明示補完された値だけを使用し、推測による接続は行わない。


## 9. source keyと冪等性

既存DBの主キー構造はProviderごとに異なるため、Sync AgentはProvider別Adapterで既存キーをcanonical JSONへ変換する。canonical JSONはプロパティ順序、日付形式、NULL表現、文字列trimを固定し、UTF-8のSHA-256を`source_key_hash_sha256`および`source_result_key_hash_sha256`として保存する。これにより、eventIdが変わった再送でも自然キーで重複を防止できる。

## 10. Providerアプリ参照境界

両ProviderのProviderアプリケーションは、出荷DB内の`FieldTest`スキーマおよび参照Viewを参照する。Sync Agentは`FieldTest`への正規登録と参照View提供までを担い、Providerアプリ側の画面、帳票、検索条件、印字制御はProviderアプリ側の実装範囲とする。

## 11. SQL Server名の診断

設定ファイルからSQL Server名が取得できない場合、または空欄・空白のみの場合、Sync Agentは`CFG-DB-001`で設定診断を失敗させる。端末名、`localhost`、`.`、既定インスタンス、サンプルDB名、他Providerの設定値を使った補完は行わない。

## 03_詳細設計書_第1分冊_基本構成・認証・同期


版数: 第1.0版

### 本書の位置づけ

本書は、共通基本原則に従い、Sync Agentの基本構成、認証、同期イベント、冪等性、Agent Leaseを定義する。Provider Adapterは読取元DBの差異を吸収し、登録先はProviderにかかわらず出荷DB内の`FieldTest`スキーマへ統一する。

## 1. 同期構成

Sync Agentは、Provider設定解決、既存DB読取、FieldTest登録、クラウド同期を1つの責務境界として実装する。Provider Adapterは読取元のDB差異だけを吸収し、登録先は両Providerとも`FieldTest`スキーマへ統一する。

## 2. 認証

|対象|方式|
|---|---|
|現場Web利用者|クラウド認証。個人アカウントと権限ロールで制御する。|
|Sync Agent|Agent credential。個人アカウントとは分離し、短命トークンでクラウドAPIへ接続する。|
|Provider DB|設定ファイルまたは資格情報ストアから取得したDB資格情報を使用する。ログには平文出力しない。|

## 3. 同期イベント

|イベント|方向|内容|
|---|---|---|
|ShipmentSnapshot|Provider DB → Cloud|予定、出荷、現場情報の同期。|
|TestResultConfirmed|Cloud → Provider DB|確定済み試験結果のFieldTest登録。|
|PhotoRegistered|Cloud → Provider DB|写真メタデータと保存先のFieldTest登録。|
|OcrResultConfirmed|Cloud → Provider DB|OCR確認結果のFieldTest登録。|
|AgentHealth|Agent → Cloud|Provider設定診断、DB疎通、Migration状態、同期状態。|

## 4. 冪等性

クラウドAPIは`eventId`を発行し、Sync Agentは`FieldTest.InboundEvent`で受信状態を管理する。登録時は`eventId`の一意制約に加え、Provider別canonical JSONから生成した`source_result_key_hash_sha256`、写真の`result_id + photo_kind + sha256`、OCR結果の`result_id + photo_id + event_id`で重複登録を防ぐ。再送時は同一内容が登録済みであれば成功扱いとし、別内容が同一自然キーに衝突する場合は競合エラーとして監査する。

## 5. 同期状態

`FieldTest.SyncState`はProvider、plantId、plantCode、cursorName、cursorValue、lastSyncedAt、status、lastErrorを保持する。通信断、DB接続断、Migration不整合は同期状態と監査に残し、Providerの自動切替は行わない。

## 6. Agent Lease

複数Agentが同一出荷DBを処理する構成では、`FieldTest.AgentLease`で排他する。`expires_at`超過leaseは同一トランザクション内で失効扱いにして再取得できる。


## 7. Inbound / Outboxイベント管理

|区分|テーブル|用途|
|---|---|---|
|受信|`FieldTest.InboundEvent`|クラウドから受信した試験結果、写真、OCR結果、設定イベントの処理状態を保持する。|
|送信|`FieldTest.OutboxEvent`|既存DB読取結果、監査、同期状態をクラウドへ送信する状態を保持する。|

InboundEventは`Received`、`Processing`、`Processed`、`Failed`、`DeadLetter`を状態として持つ。OutboxEventは`Pending`、`Sending`、`Acked`、`Failed`、`DeadLetter`を状態として持つ。いずれも`retry_count`、`next_retry_at`、`last_error_code`、`last_error_message`、`correlation_id`を保持し、通信断・一部成功・再送を安全に扱う。

## 8. Agent credential

Agent credentialは個人利用者アカウントとは分離する。tokenは短命とし、audience、tenantId、orgId、plantId、scopeを検証する。Sync Agentは画面ログイン権限を持たず、同期API、設定取得API、自動更新manifest取得APIのみを呼び出す。

## 9. Agent lease

同一Provider・同一plantのSync Agent多重起動を避けるため、DB内の`FieldTest.AgentLease`で排他する。期限切れleaseは`FieldTest.usp_AcquireAgentLease`が同一トランザクションで`released_at`を設定してから新規取得する。直接INSERTによるlease取得は禁止する。

## 04_詳細設計書_第2分冊_写真保存・ローカル参照・地図連携


版数: 第1.0版

### 本書の位置づけ

本書は、共通基本原則に従い、写真保存、OCR主写真、ローカル参照、地図連携を定義する。写真本体はBase64文字列を正本にせず、`FieldTest.TestPhoto`には保存先、ハッシュ、写真種別、OCR主写真フラグ等のメタデータを保持する。

## 1. 写真保存方針

写真本体はBase64文字列を正本にしない。クラウド側ではBlobまたはファイルストレージ、ローカル側では管理対象フォルダまたは既定保存先を使用し、`FieldTest.TestPhoto`には保存先URI、ローカルパス、SHA-256、MIME type、サイズ、撮影日時、写真種別を保持する。

## 2. OCR主写真

同一試験結果に対して`is_ocr_primary = 1`の写真は1件のみ許可する。DB側ではfiltered unique indexを使用し、アプリ制御だけに依存しない。

## 3. 写真同期

1. 現場Webで写真を登録する。
2. クラウドAPIが写真本体をストレージに保存し、ハッシュを計算する。
3. OCR対象写真はOCR処理へ送信する。
4. Sync Agentは写真メタデータと保存先を受信し、Providerにかかわらず`FieldTest.TestPhoto`へ登録する。
5. 再送時は`event_id`または`sha256 + result_id + photo_kind`で重複を防ぐ。

## 4. ローカル参照

Providerアプリケーションがローカルで写真を参照する場合は、`FieldTest.TestPhoto.storage_uri`または`local_file_path`を参照する。Sync AgentはFieldTest登録までを責務とし、Providerアプリケーション側の表示方法には依存しない。

## 5. 地図連携

現場マスターに緯度・経度が存在する場合はDTOに含める。存在しない場合は住所・現場名での検索候補に留め、Sync Agentが外部地図APIへ直接問い合わせることは必須としない。


## 7. 写真の重複防止

`FieldTest.TestPhoto`は、同一`event_id`の再送だけでなく、同一`result_id`、同一`photo_kind`、同一`sha256`の再送を重複として扱う。写真メタデータには、`storage_uri`または`local_file_path`のいずれかを必須とし、写真所在を持たないレコードは登録しない。

## 8. OCR主写真の一意性

同一`result_id`に対して`is_ocr_primary = 1`の写真は1件のみとする。OCR主写真を差し替える場合は、旧主写真の`is_ocr_primary`を0に更新したうえで新主写真を登録する。履歴として写真レコードは削除しない。

## 05_詳細設計書_第3分冊_文字読取・外部連携・画面仕様


版数: 第1.0版

### 本書の位置づけ

本書は、共通基本原則に従い、OCR対象、確認フロー、画面構成、外部連携を定義する。OCR確認結果はProvider非依存のクラウドAPIとSync Agentを経由し、出荷DB内の`FieldTest.TestResult`および`FieldTest.OcrResult`へ登録する。

## 1. OCR対象

OCRは、フレッシュ試験結果の確認補助として、黒板写真または試験結果写真から値候補を抽出する。抽出対象はスランプ、フロー、空気量、コンクリート温度、単位容積質量、塩化物量、外気温、単位水量、打設箇所、備考などとする。

## 2. 確認フロー

1. 利用者がOCR対象写真を登録する。
2. クラウドAPIがOCRを実行し、値候補と信頼度を返す。
3. 利用者が確認画面で値を修正・確定する。
4. 確定結果はクラウドに保存され、Sync Agentへ同期される。
5. Sync AgentはProviderにかかわらず`FieldTest.TestResult`と`FieldTest.OcrResult`へ登録する。

## 3. 画面構成

|画面|内容|
|---|---|
|ログイン|利用者認証と権限確認。|
|出荷予定一覧|予定日、現場、出荷予定、状態で検索する。|
|出荷実績詳細|出荷実績、配合、現場、試験登録状態を表示する。|
|写真登録|OCR対象写真、記録写真を登録する。|
|OCR確認|候補値、信頼度、手修正欄を表示する。|
|試験結果入力|フレッシュ試験値を入力・確定する。|
|保存完了|登録結果、同期状態、エラーがあれば再試行情報を表示する。|

## 4. 外部連携

OCRエンジン、写真ストレージ、クラウドAPIはProvider非依存とする。Provider固有処理はSync AgentのAdapterとDB Migrationに限定する。


## 7. 黒板・TP・品質DBとの境界

第1.0版OCR同期は、撮影済み写真からフレッシュ試験値候補を抽出し、利用者が確認した値を`FieldTest.TestResult`および`FieldTest.OcrResult`へ登録する処理である。黒板テンプレート生成、黒板レイアウトDB読取、TP採取、供試体、強度試験、品質帳票DBへの登録は第1.0版の同期処理に含めない。

## 8. 値の保持形式

OCR元文字列は`*_text`、確定数値は`*_value`として保持する。数値化できない候補は`*_text`に保持し、`*_value`はNULLとする。画面では元文字列、補正後数値、信頼度、確認者、確認日時を表示できるようにする。

## 07_詳細設計書_第5分冊_同期機能・データベース定義

FieldTest Migrationの実行先は、Provider設定から解決した出荷DBである。Labonityでは`p_出荷管理データベース名`、Libertyでは`出荷データＤＢ名`を使用する。いずれもDB名はサンプル値ではなく設定ファイル値を正とする。

|テーブル|用途|
|---|---|
|FieldTest.SchemaMigration|Migration適用履歴。|

- 両Providerとも同じDDLを使用する。
- `provider`列に`Labonity`または`Liberty`を保持する。

- `plantId`と`plantCode`は分離する。
- `event_id`の一意制約で冪等登録を保証する。
- OCR主写真はfiltered unique indexで1件に制限する。

品質系DBは第1.0版OCR同期に不要であるため、接続診断、Migration、同期カーソル、登録先に含めない。将来、TP採取・供試体・強度試験・品質帳票まで同期する場合は、別機能として対象DBを追加する。


## 7. FieldTestスキーマ固定

第1.0版ではスキーマ名を`FieldTest`固定とする。Sync Agent設定にスキーマ名変更項目は持たせず、DDL、Migration、Providerアプリ参照View、受入条件は固定名`FieldTest`を正とする。

## 8. Migration設計

Migrationは`V001_CoreSchema`、`V002_IdempotencyIndexes`、`V003_ProviderAppViews`、`V004_AgentLeaseProcedures`の単位で管理する。各Migrationは`SET XACT_ABORT ON`、明示トランザクション、既存列・型・制約・indexの検査、互換不能時の停止、`FieldTest.SchemaMigration`への記録を行う。単純な`CREATE TABLE IF NOT EXISTS`だけで既存不整合を黙認しない。

## 9. 数値項目

フレッシュ試験値は、OCR元文字列と確定数値を分離して保持する。例として`slump_text` / `slump_value`、`air_text` / `air_value`、`concrete_temperature_text` / `concrete_temperature_value`を持つ。Providerアプリは数値列を帳票・検索・範囲検査に利用できる。

## 10. 参照View

Providerアプリが直接テーブル結合を実装しなくて済むよう、`FieldTest.v_ProviderApp_TestResult`、`FieldTest.v_ProviderApp_TestPhoto`、`FieldTest.v_ProviderApp_OcrResult`、`FieldTest.v_ProviderApp_ShipmentResultStatus`を提供する。

## 19_Sync_Agent_Provider設定仕様


版数: 第1.0版

### 本書の位置づけ

本書は、Sync Agentが保持するProvider設定JSON、解決結果モデル、失敗時の扱いを定義する。入口設定ファイルまでのパスはSync Agent設定を正とし、サンプルDB名や既定パスによる推測接続は行わない。

## 1. 設定JSON例

```json
{
  "provider": "Labonity",
  "tenantId": "00000000-0000-0000-0000-000000000000",
  "orgId": "00000000-0000-0000-0000-000000000000",
  "plantId": "00000000-0000-0000-0000-000000000000",
  "plantCode": "0001",
  "labonity": {
    "libLocalXmlPath": "C:/Labonity/Config/LibLocal.xml",
    "databaseSettingFileName": "LibertyDatabaseSetting.xml"
  },
  "liberty": {
    "libLocalIniPath": "C:/Liberty/Config/LIBLocal.Ini",
    "libctrlFileName": "LIBCTRL.ini"
  },
  "photoStorageMode": "FileReference",
  "enableSync": true,
  "updateChannel": "stable"
}
```

Windowsパスをバックスラッシュで記述する場合は、JSON仕様に従い`C:\\Labonity\\Config\\LibLocal.xml`のようにエスケープする。資料内の例は可読性のためスラッシュ形式を正とする。Providerに一致しない設定ブロックは保持してよいが、実行時には参照しない。`FieldTest`スキーマ名は固定であり、設定JSONでは変更しない。

## 2. 解決結果モデル

|項目|内容|
|---|---|
|provider|`Labonity`または`Liberty`。|
|entryConfigPath|Sync Agent設定で指定された入口設定ファイルの完全パス。|
|derivedConfigPath|Labonityでは`LibertyDatabaseSetting.xml`、Libertyでは`LIBCTRL.ini`の解決後パス。|
|sqlProviderName|SQLOLEDB等。|
|sqlServerName|設定ファイルから取得したSQL Server名。空欄・空白のみの場合は`CFG-DB-001`で停止し、補完しない。|
|sqlUserId|設定ファイルまたは資格情報ストアから取得する。監査ではマスクする。|
|masterDatabaseName|Labonityは`p_マスターデータベース名`、Libertyは`共通マスタＤＢ名`。|
|shippingDatabaseName|Labonityは`p_出荷管理データベース名`、Libertyは`出荷データＤＢ名`。|
|settingsDatabaseName|Labonityは`p_共通設定データベース名`。Libertyでは`LIBCTRL.ini`のシステム設定を設定情報として扱う。|
|plantCode|LabonityはSync Agent設定またはDB値、Libertyは`初期工場コード`から導出した値を既定とする。|
|resultWriteMode|`FieldTestSchema`。両Provider共通。|
|ignoredDatabaseNames|設定ファイルに存在しても第1.0版OCR同期で接続・同期対象にしないDB名。例: 品質DB、写真台帳DB、販売DB。|

## 3. OCR同期で使用するDB

|Provider|読取DB|登録DB|使わないDB|
|---|---|---|---|
|Labonity|マスターDB、出荷管理DB|出荷管理DBの`FieldTest`スキーマ|品質管理DB、販売管理DB、動荷重DB|
|Liberty|共通マスタDB、出荷データDB|出荷データDBの`FieldTest`スキーマ|品管データDB、品管データ2DB、写真台帳DB、販売DB、受払DB|

## 4. 失敗時の扱い

設定ファイルが存在しない、文字コードが読めない、必須キーがない、SQL Server名が空欄、DBへ接続できない、Migration診断に失敗した場合は設定エラーとして停止する。Providerの推測や自動切替は行わない。SQL Server名については`localhost`、`.`、端末名、既定インスタンス、サンプル値、他Provider設定値による補完を禁止する。品質DBや写真台帳DBの未設定・未接続は、第1.0版OCR同期の停止条件にしない。


## 5. 設定エラーコード

|コード|発生条件|処理|
|---|---|---|
|CFG-001|入口設定ファイルパスが未設定または存在しない|Provider診断を失敗させ同期停止。|
|CFG-002|派生設定ファイルを解決できない|Provider診断を失敗させ同期停止。|
|CFG-003|設定ファイルの文字コードまたはXML構造が不正|Provider診断を失敗させ同期停止。|
|CFG-DB-001|SQL Server名が空欄または空白のみ|補完せず同期停止。|
|CFG-DB-002|出荷DB名またはマスターDB名が空欄|補完せず同期停止。|
|CFG-SEC-001|秘密値を安全に取得できない|同期停止。|

## 20_Libertyデータベーススキーマ対応表

### 本書の位置づけ

本書は、Liberty Providerにおける既存DB読取元と`FieldTest`登録先の対応を定義する。Liberty Providerでは`LIBLocal.Ini`から`LIBCTRL.ini`を解決し、出荷データDB内の`FieldTest`スキーマへ登録する。`SyukkaFresh`および`SyukkaFreshPicture`へSync Agentが直接書き込む仕様ではない。

## 1. Liberty ProviderのDB解決

Liberty Providerは、Sync Agent設定の`liberty.libLocalIniPath`で指定された`LIBLocal.Ini`を読み、`[端末設定] サーバーパス`から`LIBCTRL.ini`を解決する。`LIBCTRL.ini`の`共通マスタＤＢ名`を現場等の読取元、`出荷データＤＢ名`を予定・出荷の読取元および出荷データDBの`FieldTest`スキーマ登録先として使用する。

## 2. Liberty読取元

|用途|DB区分|代表DB名|代表テーブル|扱い|
|---|---|---|---|---|
|出荷予定|出荷データDB|LSYUDAT|Yotei1 / Yotei2 / Yotei3|読取|
|出荷実績|出荷データDB|LSYUDAT|Syukka1 / Syukka2 / Syukka3|読取|
|現場|共通マスタDB|LMASTER|Genba|読取|
|工場コード|LIBCTRL.ini|なし|初期工場コード|設定値|

実環境のDB名は`LIBCTRL.ini`の値を正とし、代表DB名を固定値として接続しない。

## 3. Liberty登録先

|用途|登録DB|スキーマ|テーブル|扱い|
|---|---|---|---|---|
|出荷識別対応|出荷データDB|FieldTest|SourceShipmentMap|登録・更新|
|フレッシュ試験結果|出荷データDB|FieldTest|TestResult|登録・更新|
|写真メタデータ|出荷データDB|FieldTest|TestPhoto|登録・更新|
|OCR結果|出荷データDB|FieldTest|OcrResult|登録・更新|
|同期状態|出荷データDB|FieldTest|SyncState|登録・更新|
|設定診断|出荷データDB|FieldTest|ProviderConfigSnapshot|登録・更新|

## 4. DTO変換

|FieldTest項目|Liberty由来|備考|
|---|---|---|
|provider|固定値`Liberty`|Provider識別。|
|plantCode|`初期工場コード`またはSync Agent設定|plantIdとは分離する。|
|sourceShipmentKey|出荷予定・出荷実績の複合キー|canonical JSONとSHA-256で保持する。|
|shipmentDate|予定日または出荷日|日付型へ正規化する。|
|shipmentNo|当日予定No、SEQ等|表示・検索用。|
|siteName|Genbaまたは出荷データの現場名|現場マスターが取得できない場合は出荷側名称を使用する。|
|slump / flow / air等|現場アプリ入力値またはOCR確認値|FieldTest.TestResultへ登録する。|
|photo metadata|クラウド写真情報|FieldTest.TestPhotoへ登録する。|

## 5. Providerアプリケーション連携

Liberty側アプリケーションは、出荷データDB内の`FieldTest`スキーマを参照して現場試験アプリの結果を取得する。Sync Agentは`FieldTest`登録、同期状態、監査までを担当し、Liberty側画面・帳票・参照SQLの実装には依存しない。

## 6. 既存フレッシュテーブル資料の扱い

既存フレッシュテーブル資料は、列名、試験区分、写真の持ち方、過去互換用語を理解するための参考資料として収録する。第1.0版のSync Agentは、既存フレッシュテーブルを正規登録先にしない。


## 7. Liberty source key生成

Libertyの`source_shipment_key`は、設定解決結果と既存出荷DB読取値から次のcanonical JSONを生成する。プロパティ順序、日付形式、NULL表現を固定し、UTF-8のSHA-256を`source_key_hash_sha256`へ保存する。

```json
{
  "provider": "Liberty",
  "sourceKeyVersion": 1,
  "plantId": "<GUID>",
  "plantCode": "<初期工場コードまたは出荷側工場コード>",
  "shippingDatabase": "<出荷データDB名>",
  "sourceTable": "Syukka1/Syukka2/Syukka3 または Yotei1/Yotei2/Yotei3",
  "syukkaYoteibi": "YYYY-MM-DD",
  "tozituYoteiNo": "<当日予定No>",
  "itinitiRenban": "<1日連番>",
  "syukkaSeqNo": "<出荷SEQまたはnull>"
}
```

試験結果の`source_result_key`は、上記に`sikenKubun`、`isTatewari`、`groupNo`、`dataKubun`を追加したcanonical JSONから生成する。

## 22_Provider別設定ファイル解決仕様


版数: 第1.0版

### 本書の位置づけ

本書は、Provider別の入口設定ファイルパス、派生設定ファイル、DB名解決、診断結果を定義する。Labonityは`LibLocal.xml`から`LibertyDatabaseSetting.xml`を、Libertyは`LIBLocal.Ini`から`LIBCTRL.ini`を解決する。品質系DB・写真台帳DB等は第1.0版OCR同期の必須接続先にしない。

## 1. 入口設定ファイルパス

両Providerとも、入口設定ファイルまでのパスはSync Agent設定に保持する。環境ごとのインストール先や共有フォルダ構成を推測せず、管理者が指定したパスを正とする。

|Provider|Sync Agent設定項目|入口設定ファイル|役割|
|---|---|---|---|
|Labonity|`labonity.libLocalXmlPath`|`LibLocal.xml`|サーバーフォルダを取得する。|
|Liberty|`liberty.libLocalIniPath`|`LIBLocal.Ini`|サーバーパスを取得する。|

## 2. Labonity解決仕様

Labonity Providerは、Sync Agent設定の`labonity.libLocalXmlPath`から`LibLocal.xml`を読み取る。XML要素`p_サーバーフォルダパス`の値を取得し、パス正規化、存在確認、読取権限確認を行う。次に、そのフォルダ配下にある`LibertyDatabaseSetting.xml`を読み取る。

|XML要素|Sync Agent内部項目|第1.0版OCR同期での扱い|
|---|---|---|
|`p_SQLプロバイダー名`|sqlProviderName|使用する。|
|`p_SQLサーバー名`|sqlServerName|必須。空欄・空白のみの場合は`CFG-DB-001`で停止し、補完しない。|
|`p_SQLユーザーID`|sqlUserId|使用する。監査ではマスクする。|
|`p_SQLパスワード`|sqlPasswordSecret|使用する。ログへ出さない。|
|`p_マスターデータベース名`|masterDatabaseName|使用する。|
|`p_出荷管理データベース名`|shippingDatabaseName|使用する。`FieldTest`スキーマ作成先。|
|`p_品質管理データベース名`|ignoredDatabaseNames.quality|使用しない。必須接続先にしない。|
|`p_販売管理データベース名`|ignoredDatabaseNames.sales|使用しない。|
|`p_動荷重データベース名`|ignoredDatabaseNames.dynamicLoad|使用しない。|
|`p_共通設定データベース名`|settingsDatabaseName|設定・診断用に保持する。|

## 3. Liberty解決仕様

Liberty Providerは、Sync Agent設定の`liberty.libLocalIniPath`から`LIBLocal.Ini`をCP932として読み取る。`[端末設定]`の`サーバーパス`を取得し、パス正規化、存在確認、読取権限確認を行う。次に、そのフォルダ配下にある`LIBCTRL.ini`をCP932として読み取る。

|INIキー|Sync Agent内部項目|第1.0版OCR同期での扱い|
|---|---|---|
|`ＳＱＬプロバイダー名`|sqlProviderName|使用する。|
|`ＳＱＬサーバー名`|sqlServerName|必須。空欄・空白のみの場合は`CFG-DB-001`で停止し、補完しない。|
|`ＳＱＬユーザーＩＤ`|sqlUserId|使用する。監査ではマスクする。|
|`ＳＱＬパスワード`|sqlPasswordSecret|使用する。ログへ出さない。|
|`共通マスタＤＢ名`|masterDatabaseName|使用する。現場等の読取元。|
|`出荷データＤＢ名`|shippingDatabaseName|使用する。`FieldTest`スキーマ作成先。|
|`初期工場コード`|defaultPlantCode|使用する。plantCode未指定時の既定値。|

|`販売データＤＢ名` / `受払データＤＢ名`等|ignoredDatabaseNames.other|使用しない。|

## 4. 禁止事項

- 設定ファイルが存在しない場合に既定パスを総当たりしない。
- Labonity設定からLiberty設定へ、またはLiberty設定からLabonity設定へ自動切替しない。
- サンプルDB名を固定値として接続しない。
- 品質DBや写真台帳DBを出荷DBの代替として推測利用しない。
- パスワードをログに出力しない。

## 5. 診断結果

設定解決結果には、Provider、入口設定ファイルパス、派生設定ファイルパス、更新日時、解決した出荷DB名、マスターDB名、診断状態、エラーコード、秘密値マスク状態、未使用DB名を保持する。秘密値そのものは保持しない。


## 6. SQL Server名空欄時の扱い

SQL Server名は両Providerとも必須である。空欄、空白のみ、XML要素またはINIキー欠落の場合は`CFG-DB-001`として設定診断を失敗させる。Sync Agentは、次のいずれも行わない。

- `localhost`、`.`、`(local)`、端末名、既定インスタンスへの補完。
- サンプルDB名、過去接続履歴、別ProviderのSQL Server名の利用。
- SQL Server名が空欄のままDB名だけで接続する処理。

この診断はMigrationより前に実施する。

## 27_Migration・冪等性・Lease詳細仕様


版数: 第1.0版

### 本書の位置づけ

本書は、FieldTest Migration、source key生成、再送安全、Agent lease取得手順を定義する。両Providerとも出荷DB内の固定スキーマ`FieldTest`を使用する。

## 1. Migration単位

|Migration|内容|
|---|---|
|V001_CoreSchema|SchemaMigration、ProviderConfigSnapshot、SourceShipmentMap、SyncState、TestResult、TestPhoto、OcrResult、AuditEventを作成する。|
|V002_EventQueuesAndIndexes|InboundEvent、OutboxEvent、冪等性index、写真重複防止index、OCR主写真一意indexを作成する。|
|V003_ProviderAppViews|Providerアプリ参照Viewを作成する。|
|V004_AgentLeaseProcedures|AgentLease、lease取得・解放procedureを作成する。|

各Migrationは`SchemaMigration`へ`migration_id`、`checksum_sha256`、`applied_at`、`description`を記録する。既存オブジェクトの列・型・制約・indexが互換不能な場合は、黙って続行せず`MIG-001`として停止する。

## 2. source_shipment_key

source keyは、Provider別既存キーをcanonical JSONに変換して生成する。canonical JSONは次の規則を守る。

- UTF-8でSHA-256を計算する。
- プロパティ順序を仕様書の順序で固定する。
- 日付は`YYYY-MM-DD`、日時はUTC ISO-8601に統一する。
- NULLはJSONの`null`で表す。
- 文字列は前後空白を除去し、内部空白は原則保持する。
- `sourceKeyVersion`を必ず含める。

## 3. source_result_key

source_result_keyは、source_shipment_keyのcanonical JSONに、`sikenKubun`、`isTatewari`、`groupNo`、`dataKubun`を追加して生成する。これにより、同一出荷に複数試験結果が存在しても一意に識別できる。

## 4. 冪等性

|対象|冪等条件|
|---|---|
|TestResult|`event_id`、または`provider + plant_id + source_result_key_version + source_result_key_hash_sha256`が一致。|
|TestPhoto|`event_id`、または`result_id + photo_kind + sha256`が一致。|
|OcrResult|`event_id`、または`result_id + photo_id + ocr_status + normalized_json_hash`が一致。|
|InboundEvent|`event_id`が一致。|
|OutboxEvent|`event_id`が一致。|

同一自然キーで内容が異なる場合は、上書きせず競合として監査する。上書きが必要な運用は、Providerアプリまたは管理機能の明示操作として扱う。

## 5. Agent lease

Agent leaseは、`FieldTest.usp_AcquireAgentLease`で取得する。procedureは同一トランザクション内で、期限切れactive leaseに`released_at`を設定し、同じ`resource_name`のactive leaseが存在しないことを確認して新leaseを挿入する。`UPDLOCK`と`HOLDLOCK`で競合を抑止する。直接INSERTによるlease取得は禁止する。

## 6. SQL Server名空欄時の停止

設定診断においてSQL Server名が空欄の場合は、DB接続、Migration、lease取得の前に停止する。補完、推測、fallbackは行わない。

CREATE PROCEDURE FieldTest.usp_AcquireAgentLease
    @agent_id uniqueidentifier,
    @resource_name nvarchar(200),
    @lease_seconds int = 300
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON;

    DECLARE @now datetime2(3) = SYSUTCDATETIME();
    DECLARE @expires_at datetime2(3) = DATEADD(SECOND, @lease_seconds, @now);

    BEGIN TRANSACTION;

    UPDATE l WITH (UPDLOCK, HOLDLOCK)
        SET released_at = @now
    FROM FieldTest.AgentLease l
    WHERE l.resource_name = @resource_name
      AND l.released_at IS NULL
      AND l.expires_at < @now;

    IF EXISTS (
        SELECT 1
        FROM FieldTest.AgentLease WITH (UPDLOCK, HOLDLOCK)
        WHERE resource_name = @resource_name
          AND released_at IS NULL
    )
    BEGIN
        ROLLBACK TRANSACTION;
        THROW 51001, 'Active agent lease already exists.', 1;
    END;

    INSERT INTO FieldTest.AgentLease(agent_id, resource_name, acquired_at, expires_at)
    VALUES(@agent_id, @resource_name, @now, @expires_at);

    COMMIT TRANSACTION;
END;
GO

CREATE PROCEDURE FieldTest.usp_ReleaseAgentLease
    @agent_id uniqueidentifier,
    @resource_name nvarchar(200)
AS
BEGIN
    SET NOCOUNT ON;
    UPDATE FieldTest.AgentLease
       SET released_at = SYSUTCDATETIME()
     WHERE agent_id = @agent_id
       AND resource_name = @resource_name
       AND released_at IS NULL;
END;
