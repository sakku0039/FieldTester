## 3. 構成要素と責任分担

| **構成要素**                 | **担う責任**                                                  | **担わない責任**                              |
|------------------------------|---------------------------------------------------------------|-----------------------------------------------|
| 現場試験Web                  | 個人ログイン、予定・出荷参照、写真登録、地図、処理状態の表示  | 基幹データ編集、TP正式入力、OCR結果の確定     |
| 現場試験アプリ用クラウド環境 | 権限確認、写真保管、OCR、処理状態、同期イベント、監査         | 社内DBへの直接接続、既存Labonityの正式保存    |
| Sync Agent                   | 参照データ同期、写真・OCR結果取得、JPEG保存、状態・監視       | 個人代理ログイン、既存Labonity向けローカルAPI |
| 既存Labonity                 | ローカル写真・OCR候補の確認、反映先検証、入力欄反映、通常保存 | クラウド認証、OCR実行依頼、クラウドAPI呼出し  |
| Liberty Account              | 人間とAgentの認証、組織・工場・権限範囲の提供                 | 写真・OCR・業務データの保管                   |

## 4. 配置・接続方針

| **配置先**           | **構成**                                     | **接続・留意点**                                                  |
|----------------------|----------------------------------------------|-------------------------------------------------------------------|
| スマートフォン       | 現場試験Web                                  | HTTPSで認証・クラウドへ接続。カメラ、ライブラリ、外部地図を利用。 |
| クラウド             | Web/API、写真保管、OCR、同期イベント、監査   | 組織・工場ごとにデータを分離し、写真・OCRを正本として管理。       |
| Labonityサーバー     | Sync Agent、SQL Server、既存設定、写真出力先 | Windows Serviceとして常駐し、クラウドへアウトバウンドHTTPS。      |
| Labonityクライアント | 既存デスクトップアプリ                       | SQL Serverと共有写真を参照。クラウドへ直接接続しない。            |

- 同一工場で有効なSync Agentは原則1台とし、二重同期を防止する。

- 複数クライアントで写真を参照する場合は、共通の共有パスを使用する。

- DB接続は既存設定を起点に解決し、同じ情報を別設定へ二重入力しない。

## 5. 主要処理フロー

![図2_写真登録から正式保存まで](%E7%94%BB%E5%83%8F/02_%E5%9F%BA%E6%9C%AC%E8%A8%AD%E8%A8%88%E6%9B%B8/%E5%9B%B32_%E5%86%99%E7%9C%9F%E7%99%BB%E9%8C%B2%E3%81%8B%E3%82%89%E6%AD%A3%E5%BC%8F%E4%BF%9D%E5%AD%98%E3%81%BE%E3%81%A7.png)

図2 写真登録から正式保存まで

### 5.1 写真登録

1.  対象日の出荷実績を開き、現場・時刻・車番を確認する。

2.  写真用途（通常／OCR黒板）と取込方法（カメラ／ライブラリ）を選ぶ。

3.  対象出荷、プレビュー、画質警告を確認して登録する。

4.  クラウドで写真を確定し、OCR黒板の場合だけOCR処理を開始する。

### 5.2 同期とOCR

5.  OCRが値、信頼度、画質、警告、検証結果を保存する。

6.  クラウドが写真・OCR結果の同期対象を発行する。

7.  Sync Agentが結果を取得し、社内DBと出荷別JPEGを更新する。

8.  処理済み位置を記録し、停止後は続きから再開する。

### 5.3 既存Labonityへの反映

9.  TP、行番号、データ区分、出荷実績を画面上で確定する。

10. 社内DBから現在有効なOCR結果と原本写真を取得する。

11. 原本、OCR候補、現在値、信頼度、警告を同じ画面で比較する。

12. 担当者が確認・修正し、入力欄へ反映する。

13. 既存の通常保存処理で正式登録し、監査を記録する。

## 6. データ設計方針

| **区分**       | **正本・配置**                          | **主なデータ**                                      |
|----------------|-----------------------------------------|-----------------------------------------------------|
| 既存業務データ | MsDat                                   | 現場名、住所、座標など                              |
| 既存業務データ | NaDat                                   | 出荷予定、出荷実績など                              |
| 既存業務データ | ExDat                                   | TP採取結果、フレッシュ試験、出荷との関連            |
| クラウド正本   | クラウドDB・写真保管                    | 参照コピー、写真、OCR、処理状態、同期イベント、監査 |
| 社内派生データ | 出荷管理DBのFieldTestスキーマ・ファイル | 写真参照、JPEG、OCR参照、反映監査、同期状態         |

- 全連携データに組織・工場を識別する情報を持たせる。

- 写真本体はDBに格納せず、写真保管とメタデータを分離する。

- 既存列の追加を避け、連携用テーブルを追加する。

- イベント番号、ハッシュ、重複防止キーで再送・順序逆転に対応する。

## 7. 認証・認可

| **主体**      | **識別**         | **認可の軸**                           | **利用範囲**    |
|---------------|------------------|----------------------------------------|-----------------|
| 現場Web利用者 | 個人アカウント   | 組織、サービス利用権限、役割・操作権限 | 現場Web向けAPI  |
| Sync Agent    | 専用Agent ID     | 組織、工場、同期専用権限、有効状態     | 同期向けAPIのみ |
| 既存Labonity  | クラウド主体なし | 既存のローカルユーザー・OS・DB権限     | クラウドAPIなし |

- URL、トークン、DB上の組織・工場が一致することを検証する。

- Agentは短時間有効なアクセストークンを使用し、個人のパスワードや更新トークンを保存しない。

- Agentの秘密情報はOS保護領域に保存し、失効・再発行・更新ができるようにする。

- 人間操作とAgent操作を別の主体として監査する。

## 8. OCR安全設計

| **観点**   | **設計方針**                                                 |
|------------|--------------------------------------------------------------|
| 対象       | OCR取込用黒板写真だけを対象とし、通常写真は処理しない。      |
| 説明可能性 | 項目別信頼度、画質、警告、候補値、検証理由を保存・表示する。 |
| 反映先確認 | TP、行番号、データ区分、出荷実績の一致を確認する。           |
| 妥当性確認 | 型、桁数、単位、範囲、車番、現在値との差分を確認する。       |
| 人の確認   | 信頼度が高くても自動確定・自動上書きを行わない。             |
| 正式保存   | OCRは入力欄への反映まで。DB保存は既存の通常保存とする。      |
| 監査       | OCR候補、確定値、修正、保存結果、処理モデルの版を追跡する。  |

## 9. 障害・オフライン

| **事象**               | **基本対応**                                                                         |
|------------------------|--------------------------------------------------------------------------------------|
| 現場の通信断           | 取得済みの出荷では端末キューから再送。出荷未同期時は標準カメラで撮影し、後から登録。 |
| クラウド一時障害       | 写真・OCR・同期処理を再試行し、重複防止キーで二重登録を防ぐ。                        |
| Agentの設定不備        | 認証、DB、権限、既存設定が不正な場合は安全側で同期を開始しない。                     |
| DB・共有フォルダ一時断 | サービスを維持し、待機状態から再試行して処理済み位置から再開する。                   |
| 写真変換失敗           | 写真生成ジョブを失敗・再試行状態として追跡し、管理者が再出力できるようにする。       |
| 容量不足               | 警告後、既存ファイルを勝手に削除せず、新規出力を安全側で停止する。                   |
| 二重Agent              | ロック情報により片方を停止し、管理者操作で切り替える。                               |

## 11. セキュリティ・監査

- 写真の直接公開URLは使用せず、短時間有効な一時URLを使用する。

- 組織・工場・利用主体・権限範囲を各要求で確認する。

- 秘密情報、アクセストークン、SQLパスワードをログや診断画面に表示しない。

- 削除は論理削除を先行し、承認済みの保持期間後に物理削除する。

- OCR修正値の外部学習利用は初期状態で無効とする。

- 監査には主体、組織、工場、対象、結果、時刻、追跡番号を記録する。

## 15. 設計の優先順位

- 安全な業務継続：OCRや同期が使えなくても手入力を継続できる。

- 誤反映防止：出荷、TP、renban、datakubunを検証し、人が原本を確認する。

- データ分離：orgId、plantId、agentId、tenant_idをAPI・DB・ファイルで一貫して検証する。

- 既存Labonityとの疎結合：クラウドAPIやIPCを直接呼ばず、NaDatとローカル写真を参照する。

- 運用可能性：認証、DB、共有フォルダ、容量、滞留、二重稼働を監視・診断できる。

## 16. ローカルDB配置と設定解決

新しい物理データベースは作成せず、既存の出荷管理DBに専用の FieldTest スキーマを追加します。出荷管理DB名は既存の DB設定XMLから取得し、スキーマ名は Sync Agent設定で明示します。

| 対象     | 方式                           | 補足                                     |
|----------|--------------------------------|------------------------------------------|
| DB名     | p_出荷管理データベース名を使用 | 通常 NaDat。環境値に追従する。           |
| 既存領域 | \[dbo\] スキーマ               | 予定・出荷テーブルは変更しない。         |
| 追加領域 | \[FieldTest\] スキーマ         | ローカル連携12テーブルを配置する。       |
| XML変更  | 不要                           | 新しいDB項目は追加しない。               |
| 起動確認 | 必須                           | DB・スキーマ・テーブル・権限を検証する。 |
